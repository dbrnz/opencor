The following limitations have been observed while using OpenCOR on our supported platforms. They may apply to you, or not, depending on the platform you use and your specific circumstances.

Windows, Linux and OS X
  • By default, OpenCOR uses your system's language for menus, dialog boxes, etc., as long as it is either English or French (please contact us if you would like OpenCOR to support other languages). If your system uses another language, then OpenCOR will default to English. Otherwise, if you specifically specify English or French, then please be aware that system messages, dialog boxes, etc. will still be displayed using your system's language (assuming it is not one of the languages supported by OpenCOR).
  • OpenCOR uses the CellML API, which is known to have the following limitations:
    • It will crash OpenCOR if you try to export a CellML file to a user-defined format that is described in a file that contains valid, but unknown, XML.
    • It may incorrectly (in)validate certain CellML files.
Windows
  • OpenCOR's FileBrowserWindow plugin may, on some systems, result in OpenCOR being slow to respond at startup. This has nothing to do with OpenCOR, but most likely something to do with a Windows shell add-on that is installed on your system. This page may help you address the issue, but if not then you might have to unselect the FileBrowserWindow plugin.
Linux
  • Upon starting, the position of the main window may be slightly different from the one in your previous session. This is due to a lack of compliancy between window managers on Linux. So, if you can, try another window manager.
OS X
  • If you run a simulation on OS X 10.9 (Mavericks), then the first time you use a particular version of a solver, the simulation will be 'slow'. However, subsequent runs will be 'fast'. It looks like it might have nothing to do with OpenCOR since all simulations are 'fast' on Windows and Linux, as well as on Mac OS X 10.7 (Lion) and OS X 10.8 (Mountain Lion).
